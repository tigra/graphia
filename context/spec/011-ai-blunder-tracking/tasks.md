# Tasks: AI Blunder Tracking (Spec 011)

Vertical slices for [spec 011](./functional-spec.md) per its
[technical-considerations](./technical-considerations.md). The deliverable is a
make-gated measurement harness (`src/graphia/tools/blunder_eval.py`,
`make blunder-eval`) and a repo-committed quality ledger
(`evals/blunder-ledger.yaml`) — **measurement only, zero production-code change**
(functional-spec [§2.4 "Measurement only — the game itself is unchanged"](./functional-spec.md)).

Each slice leaves the harness runnable and grows the ledger record; offline
verification is `uv run pytest -q` (the harness reaches a real model, so — like
`eval-dialogue` and `ollama-smoke` — it lives outside `pytest`; the mocked suite
never runs it). Slice 5's live run against both providers is the
[§2.2](./functional-spec.md)/[§2.3](./functional-spec.md) acceptance walk and is
**user-run** (needs the verified Ollama pair installed and, for Bedrock, live AWS
credentials + real tokens). Agents: `langgraph-agentic` (driver, detectors,
capture proxy), `python-backend` (CLI, provenance, ledger, docs), `testing`
(offline suite).

## Notes for the implementer

- **No fix, no guard.** This spec only *counts*. If the first records show a blunder is frequent, the fix (prompt nudge, mechanical guard) is a **later** evidence-driven spec measured against this baseline — out of scope here (functional-spec [§2.4](./functional-spec.md), Out-of-Scope).
- **No reader/comparison tooling.** The ledger is the deliverable; a console table viewer or regression gate is a possible later increment (it would add the YAML *parser* dependency this increment deliberately avoids).
